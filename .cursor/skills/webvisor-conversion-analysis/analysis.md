# Анализ визитов лендинга

Методология анализа визитов. Секции, milestones, цель конверсии и стадии воронки определяются на **этапе 0** из сайта и JSON-данных.

## Контекст

**Входы:** `site_url`, `data_dir`, `site_dir`, `output_dir` (см. `../README.md`).

**Цель:** найти системные UX-причины неконверсии. Итог анализа — полный `hypotheses.md` по `report-template.md`; внедрение — этапы 4.5–7 в `implementation.md`.

**Формулировка гипотез:** см. `hypothesis-format.md`.

---

## Этап 0. Карта лендинга → `landing-map.json`

Сохранить в `{output_dir}/landing-map.json` **до** разбора визитов.

### Обязательный DOM-аудит

1. **Открой `site_url`** (браузер / fetch / MCP) и пройди страницу сверху вниз.
2. Зафиксируй **реальные секции** (`section`, `header`, блоки с заголовками) — границы `yStart`/`yEnd` из DOM или scroll-аудита, **не** деление pageHeight на равные части.
3. У **каждой** секции заполни `observed`: всё заметное из DOM (заголовки, текст, интерактив, медиа) — **без** заранее заданного списка «что искать».
4. Дополни из JSON: `pageHeight`, `inputNodes`, `formEvents`, паттерны `clicks[].selector`, submit у конвертированных.

**Запрещено:** секции «Блок 1 (y 0–851)» без DOM; пустой `observed: []`; milestones только из арифметики без привязки к блокам страницы.

### Structural audit (наблюдения этапа 0)

Зафиксируй в `landing-map.json` → поле `structuralAudit` (массив строк-фактов) и дублируй ключевое в `observed` секций.

**Правила:**

- Только **факты** из прохода по DOM и JSON, сформулированные своими словами.
- **Не** используй заранее заданный чеклист тем — фиксируй то, что **реально увидел** на этой странице.
- Не формулируй как рекомендации («нужно добавить…»).

```json
"structuralAudit": [
  "s1: на первом экране видно …",
  "Блок s4 (y=…): впервые появляется …",
  "…"
]
```

Если при аудите нашёл элементы, которые **выглядят интерактивными, но не реагируют** на действие — добавь селекторы в `falseAffordances` (только из DOM, не из догадок).

### Структура landing-map

```json
{
  "siteUrl": "https://example.com/",
  "pageHeight": 8000,
  "sections": [
    {
      "id": "s1",
      "label": "Заголовок h2/h1 из DOM",
      "yStart": 0,
      "yEnd": 900,
      "observed": ["h1", "кнопка …", "перечисление фактов DOM"]
    }
  ],
  "conversionGoal": {
    "type": "form_submit",
    "formSelector": "селектор из inputNodes / DOM",
    "fieldSelectors": ["из inputNodes"]
  },
  "falseAffordances": [".selector-from-dom-audit"],
  "milestones": [
    { "id": "m1", "y": 4000, "label": "Описание блока из DOM (не «50% scroll» без контекста)" },
    { "id": "m2", "y": 7200, "label": "Блок, соответствующий conversionGoal (из DOM / JSON)" }
  ]
}
```

**Правила:**

- Id секций и milestones — **нейтральные** (`s1`, `m1`, …).
- `milestones[]` — блоки, **обнаруженные** при аудите; label описывает содержание блока, не только координату.
- `falseAffordances` — селекторы из DOM-аудита, если элемент не реагирует на действие.
- `observed` — факты DOM, не список «что исправить».

### `section_at(y)`

Секция, где `yStart ≤ y < yEnd`.

## Этап 1. Воронка

Построй воронку по всем визитам. Стадии — id milestones из landing-map + служебные:

| Стадия | Критерий |
|--------|----------|
| `load` | визит начался |
| `first_viewport` | maxScrollY ≥ высота viewport |
| `m1`, `m2`, … | maxScrollY ≥ milestone.y |
| `interaction_start` | focus/input на полях conversionGoal |
| `converted` | submit или целевое действие |

**Главный отвал** — переход с максимальным падением N.

---

## Этап 2. Группы неконвертированных

Группируй по **id последней достигнутой стадии** воронки.

Для каждой группы добавь **человекочитаемое описание** (для `hypotheses.md`) по **label milestone** и evidence, например:

- «Не дошёл до …» (блок из milestone label)
- «Достиг …, не выполнил следующее ожидаемое действие»
- «Начал interaction по conversionGoal, не завершил»

Сравни converted vs non-converted: `durationSec`, scroll, клики, `section_at(maxScrollY)`.

### Профиль конвертированных визитов

Разбери **каждый** конвертированный визит вручную (их обычно мало):

- хронология scroll / клики / formEvents;
- какие секции и milestones прошёл;
- чем путь отличается от типичного неконвертированного **на той же глубине scroll**.

Включи вывод в `hypotheses.md` → раздел «Профиль конвертированных».

### Доминирующий поздний кластер (>70% неконвертированных на одной стадии)

Если большинство отваливается на **одной поздней стадии** с **похожим scroll-path** (глубина, секции):

1. Сравни неконвертированных этой группы с конвертированными при **сопоставимом** maxScrollY.
2. Если JSON-поведение похоже, а конверсия низкая — **primary одного визита не объясняет весь кластер**.
3. Ищи **накопительные** факторы: длительность в контентных секциях, контекст до точки конверсии, structural audit (этап 0).
4. Для части визитов кластера добавь **secondary** (см. `hypothesis-format.md`), если `durationSec` высокий или длинный путь через контентные секции.

**Анти-паттерн:** не своди 80%+ визитов к одной формулировке «проблема только у точки конверсии», если карта и converted-профиль указывают на факторы выше по странице.

## Этап 3. Разбор визитов

Один агент / задача на JSON-файл (исключая `_summary.json`, `manifest.json`).

Каждый получает: `landing-map.json`, один визит, `hypothesis-format.md`.

**На выход:** `{output_dir}/work/visits/<visitId>.json` — структура из `hypothesis-format.md`.

**Обязательно:** разобрать **все** файлы визитов в `data_dir`.

---

## Этап 4. Агрегация → `hypotheses.md`

Следуй **`report-template.md`** — все разделы обязательны.

1. Кластеризуй primary по похожим `label` + `funnelStage` + `section`.
2. N и % от **неконвертированных**.
3. Топ-кластеры: 2–3 visitId, **suggestedFix**, секция landing-map.
4. Блок **Рекомендации** — **два слоя** (abstract UX, без кода):
   - **По кластерам** — минимум топ-3 из таблицы гипотез;
   - **По structural audit** — минимум один пункт на наблюдение, связанное с путём к конверсии (даже если N в воронке мало).
5. **Приложение** — строка на каждый визит.
6. **Сверка N:** сумма по группам = неконвертированные; топ-кластеры согласованы с группами.

### Язык в отчёте

| Где | Как писать |
|-----|------------|
| `label`, заголовки рекомендаций | Продуктовый язык, без `m1`/`m2`/`at_m3_…` |
| Колонка funnelStage | Технический id допустим |
| Per-visit JSON | `funnelStage` — id; `label` — человеческий |

Детализация под `{site_dir}` — в `implementation.md` (этапы 4.5–7).

## Критерии качества

- DOM-аудит и structural audit на этапе 0 выполнены; `observed` не пуст.
- Профиль конвертированных визитов описан.
- При доминирующем позднем кластере — вывод про несколько рычагов, не только секция конверсии.
- Сначала воронка, потом гипотезы.
- Primary не дублируют одно и то же у 90%+ визитов **разными словами** (один label на кластер — ок; разные названия одного и того же — нет).
- Evidence из JSON-полей.
- Рекомендации в двух слоях: кластеры + structural audit.
- `hypotheses.md` полный по шаблону, не сокращённый summary.

## Поля JSON визита

| Поле | Смысл |
|------|-------|
| `maxScrollY`, `scrollDepthPct` | глубина |
| `durationSec` | длительность |
| `clicks[].selector` | цель клика |
| `formEvents` | focus, input, submit |
| `inputNodes` | поля на странице |
| `pageHeight` | высота страницы |
