# Как формулировать гипотезы

Гипотеза — **правдоподобное объяснение**, почему конкретный визит не завершился конверсией. Формулируй сам, опираясь на evidence из JSON и карту лендинга.

## Структура primary-гипотезы

```json
{
  "label": "Человеческое описание причины отвала (без id milestone в тексте)",
  "behavior": "Что делал посетитель (scroll, клики, паузы, interaction с полями conversionGoal)",
  "friction": "Какой барьер, по твоей версии, помешал следующему шагу",
  "section": "id секции из landing-map",
  "funnelStage": "технический id последней стадии из воронки (например before_m1, interaction_start)",
  "evidence": "maxScrollY, selector, formEvents, durationSec — только факты",
  "suggestedFix": "Кратко: что улучшить на UX-уровне (без кода и без id/class кейса)"
}
```

**Label** — для людей в `hypotheses.md`. **funnelStage** — для воронки и группировки.

Не используй id milestone в `label` — только в `funnelStage`.

Поле `suggestedFix` — опционально на этапе 3; обязательно в агрегации топ-кластеров.

При агрегации группируй по **похожим label + funnelStage + section**.

## Как выбирать primary (одна на визит)

1. Определи **последнюю стадию воронки** из `landing-map` + JSON визита.
2. Восстанови **хронологию**: scroll → секции; клики → селекторы; formEvents → поля.
3. Сравни с **конвертированными** визитами: на каком шаге путь расходится?
4. Сформулируй **одну** главную причину — label **продуктовым языком**, не id milestone.

## Что писать в behavior и friction

- **behavior** — последовательность действий и где остановился.
- **friction** — «ожидаемый шаг → не произошёл → барьер в секции X».

Выводы только из evidence.

## Secondary (до 2)

Только при **независимом** сигнале, не дублирующем primary.

**Обязательно secondary**, если визит в **доминирующем позднем кластере** (>70% неконвертированных на одной стадии) **и**:

- `durationSec` выше медианы группы, или
- длинный путь через контентные секции до точки конверсии (по scroll / кликам).

Secondary описывает **накопительный** фактор из structural audit или converted-профиля, не дублируя primary.

## Анти-паттерны

- Не считай клики по `html` / `body` UX-проблемой.
- Не приписывай одному визиту несколько primary.
- Не выдумывай мотивы без данных.
- Различай «не достиг стадии X» и «достиг X, но не выполнил Y».
- **Не используй milestone-id в label** — только в `funnelStage`.
- Если >80% primary в одном кластере — не своди причину к одной секции; добавь secondary там, где есть сигнал накопления.

## Агрегация в отчёт

1. Объедини похожие формулировки в кластеры.
2. N и **% от неконвертированных**.
3. Для топ-кластеров: 2–3 visitId, секция, **suggestedFix**, evidence.
4. Сверь N кластеров с N групп воронки (`report-template.md`).
