# Внедрение улучшений лендинга

Этапы после анализа Webvisor (0–4): discovery ограничений в `{site_dir}`, рекомендации, правки кода, проверка.

Skill **не содержит** правил конкретного лендинга. Все ограничения — только из инструкций внутри `{site_dir}`.

---

## Этап 4.5. Discovery инструкций → `site-constraints.md`

**Обязательно до этапов 5–7.** Без этого артефакта правки в `{site_dir}` запрещены.

### Сканирование `{site_dir}`

Найди и прочитай **все** файлы с инструкциями:

1. `*.md` в корне (`README.md`, `STUDENTS.md`, `CONSTRAINTS.md`, `AGENTS.md`, …)
2. `docs/**/*.md`
3. `.cursor/rules/**` (если есть)

Не ограничивайся одним файлом. Если в разных документах есть противоречия — зафиксируй в `site-constraints.md` и выбери более строгое ограничение; при неясности — не внедряй спорное изменение.

Зафиксируй в `site-constraints.md` также **workflow после правок** из инструкций site_dir (redeploy, повторный прогон, сравнение конверсии — если описано).

### Артефакт `{output_dir}/site-constraints.md`

```markdown
# Ограничения из site_dir

## Прочитанные файлы
- path/to/file.md — кратко, что содержит

## Можно менять
(пересказ / цитаты из site_dir)

## Нельзя менять
(пересказ / цитаты из site_dir)

## Какие файлы/папки редактировать
(если указано в инструкциях)

## Workflow после правок (из site_dir)
(redeploy, симулятор, сравнение CR — если есть в инструкциях)

## Блокеры
- [ ] Инструкций не найдено → внедрение заблокировано
```

Если **ни одного** markdown-файла с правилами редактирования не найдено:

- Заполни `site-constraints.md` с пометкой «инструкции не обнаружены».
- Выполни этап 5 (рекомендации), но **не** этап 6 (правки кода).

## Этап 5. Рекомендации → `recommendations.md`

На основе **обоих слоёв** из `hypotheses.md`:

- **A.** топ-3 кластера;
- **B.** structural audit.

Для каждой рекомендации:

| Поле | Содержание |
|------|------------|
| `layer` | `cluster` / `structural` |
| `cluster` | Ссылка на кластер / N (для layer=cluster) |
| `hypothesis` | Кратко, из hypotheses.md |
| `change` | Что изменить на лендинге (UX, без кода) |
| `targetSection` | id секции из landing-map |
| `files` | Файлы из `{site_dir}` |
| `changeType` | `copy` / `css` / `layout` / `js` / … |
| `constraintRef` | Пункт из `site-constraints.md` |
| `status` | `approved` / `blocked` / `deferred` |
| `implementOrder` | порядок на странице: 1 = выше, … |

**Constraint gate:** `approved` только если разрешено в `site-constraints.md`.

**Обязательный охват:**

- все **approved** пункты из слоя A (топ-3), если не blocked;
- минимум один **approved** пункт из слоя B;
- **не откладывать** пункт только из-за малого N в воронке, если он из structural audit и approved.

**Анти-паттерн:** внедрять только правки в секции конверсии, игнорируя approved structural и ранние кластеры.

Структура `recommendations.md`:

1. Executive summary (approved / blocked / deferred)
2. Таблица рекомендаций с `implementOrder`
3. Заблокированные и отложенные — почему
4. План внедрения (см. этап 6)

## Этап 6. Внедрение в `{site_dir}`

Только если `site-constraints.md` заполнен и есть approved-рекомендации.

### Порядок правок (не по N, а **сверху страницы вниз**)

1. **Structural audit + ранние секции** (`implementOrder` 1–2): контекст до точки конверсии.
2. **Средние кластеры** (малый N, но page-level эффект): ложная кликабельность, порядок блоков.
3. **Секция конверсии** (доминирующий кластер): последней, когда контекст выше уже улучшен.

### Правила

1. Редактируй только разрешённое в `site-constraints.md`.
2. Минимальный diff: одна рекомендация — один логический commit изменений.
3. После CSS/JS правок: интерактив из `landing-map.observed` и `falseAffordances` **ведёт себя как задумано** (клик → ожидаемый эффект; декоративные элементы не имитируют клик без отклика).
4. Не трогай запретные зоны — зафиксируй компромисс в `recommendations.md`.
5. Статус каждой рекомендации: `implemented` / `partial` / `failed`.

## Этап 7. Проверка → `constraints-check.md`

```markdown
# Проверка

## Чеклист «нельзя менять»
(каждый пункт site-constraints.md → pass / fail)

## Structural audit — закрыто?
| наблюдение | рекомендация | status |

## Рекомендации
| id | layer | status | notes |

## Интерактив после правок
| элемент из observed / falseAffordances | ожидание | pass/fail |

## Изменённые файлы
- path — что изменено

## Workflow (из site_dir)
- [ ] Redeploy / повторный прогон / сравнение CR — что требует инструкция

## Итог
- [ ] Все запреты соблюдены
- [ ] Топ-3 + минимум 1 structural — implemented или объяснён defer
```

При `fail` — откати или исправь до завершения.

---

## Связь с analysis (этап 4)

`hypotheses.md` уже содержит рекомендации в двух слоях (`report-template.md`). Этапы 5–6 конкретизируют их под `{site_dir}` и **внедряют все approved**, не только доминирующий кластер.
